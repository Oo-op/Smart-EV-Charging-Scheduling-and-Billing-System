# 设计说明

# 智能充电桩调度计费系统设计说明

## 一、系统定位与总体目标

本系统面向校园智能充电站场景，目标是实现 “用户充电服务 \+ 智能调度 \+ 分时计费 \+ 后台管理” 的完整闭环。

用户通过客户端完成注册、车辆绑定、充电请求、状态查询、提前结束充电、账单查看和支付；

管理员通过后台完成充电桩状态监控、故障处理、计费规则维护和运营数据查看；

车辆 BMS 在电池充满时向系统发送充满信号，触发自动结束充电流程。

因此系统设计围绕两条主线展开：

第一条是调度主线，即用户提交请求后如何进入队列、分配充电桩、处理故障和重新调度；

第二条是计费主线，即充电开始后如何记录会话、按分时电价和服务费计算费用、生成账单并完成支付。

## 二、总体架构设计

系统采用基于 Spring Boot 的分层架构，整体划分为表现层、控制层、业务逻辑层、领域模型层、数据访问层和数据存储层。

> 表现层负责用户端和管理员端界面交互，包括登录注册、充电申请、排队状态查看、账单支付和后台监控等功能。
> 
> 控制层由 Controller 组成，负责接收 HTTP 请求、完成参数校验并调用对应业务服务。
> 
> 业务逻辑层由 Service 组成，是系统核心，负责充电申请、调度、计费、充电桩管理、故障处理等业务规则。领域模型层保存系统核心业务对象，如 User、Vehicle、ChargingRequest、ChargingPile、ChargingSession、Bill 等。数据访问层由 Repository 组成，负责数据库读写。
> 
> 数据存储层使用 MySQL 保存用户、车辆、请求、队列、充电桩、会话、账单、支付记录和故障记录。
> 
> 

系统运行链路为：

用户/管理员请求 →  Controller 接收请求 →  Service 执行业务规则 → Domain 对象完成状态变化 →  Repository 持久化数据 → Controller 返回处理结果

这种分层方式可以降低模块耦合度，使用户管理、充电调度、计费结算、充电桩管理和后台监控能够相对独立开发，同时便于后续扩展新的充电站、更多充电桩或新的计费规则。

## 三、核心业务对象设计

系统中的核心业务对象可以分为五组。

第一组是参与者对象，包括 User、Admin、Vehicle。User 表示普通用户，可绑定多辆车；Vehicle 表示车辆信息，包括车牌号、车型、电池容量等；Admin 表示后台管理员，用于运营管理和故障处理。

第二组是资源对象，包括 ChargingStation、ChargingPile、ParkingSpace。ChargingStation 表示充电站整体；ChargingPile 表示具体充电桩，分为快充桩和慢充桩；ParkingSpace 表示充电桩对应的停车位。系统默认包含 2 个快充桩、3 个慢充桩，每个充电桩对应 4 个停车位。

第三组是调度对象，包括 ChargingRequest、WaitQueue、QueueItem。ChargingRequest 表示用户提交的充电申请，保存用户、车辆、充电模式、目标电量、状态、创建时间等信息。WaitQueue 按快充和慢充分为两条逻辑队列。QueueItem 表示队列中的一个排队项，用于记录排队号、请求时间、预计充电时长和优先级等信息。

第四组是充电执行对象，包括 ChargingSession、MeterReading、ChargingPileStatus。ChargingSession 表示一次实际充电会话，记录开始时间、结束时间、初始电表读数、最终电表读数、已充电量、目标电量和会话状态。MeterReading 表示电表读数，用于计算实际充电量。

第五组是计费结算对象，包括 BillingTariff、TimeSlotRate、Bill、Payment。BillingTariff 表示计费规则，聚合多个 TimeSlotRate。TimeSlotRate 表示分时电价时段。Bill 表示账单，包括电费、服务费、总费用和费用明细。Payment 表示支付记录，保存支付方式、支付时间、交易流水和支付状态。

## 四、主要功能模块设计

系统可划分为用户管理模块、充电申请模块、调度模块、充电桩管理模块、计费结算模块、故障处理模块和后台监控模块。

用户管理模块负责用户注册、登录和车辆绑定。用户注册时系统创建 User 对象，绑定车辆时创建 Vehicle 对象，并建立 User 与 Vehicle 的关联。发起充电请求前，系统需要确认用户身份和车辆信息有效。

充电申请模块负责创建、修改和取消充电请求。用户选择车辆、充电模式和目标电量后，系统创建 ChargingRequest，并根据充电模式进入快充或慢充等待队列。如果用户仍在等候区，可以修改目标电量或取消请求；如果已经进入充电区，则不能增加充电量，只能提前结束充电后重新排队。

调度模块是系统核心。系统在充电桩空闲、用户提交请求、用户修改请求、充电桩故障恢复等事件发生时触发调度。基础调度策略是按快充/慢充分队列管理，并从对应队列中选择合适车辆分配到空闲桩。完整策略采用“预计完成时间最短优先”，即遍历队列中车辆，计算其在当前空闲桩上的预计完成时间，优先调度充电需求较小、完成时间较早的车辆。若实现复杂度受限，可使用 FCFS 作为简化策略，但应在说明中注明。

充电桩管理模块负责充电桩上电、启动服务、关闭、参数设置和状态监控。充电桩状态包括空闲、充电中、故障、维护、离线等。管理员可以查看各充电桩实时状态，系统也可以通过硬件适配器读取功率、电压、电流、故障码等实时数据。

计费结算模块负责充电会话记录、费用计算、账单生成和支付。充电开始时，系统创建 ChargingSession，记录开始时间和初始电表读数；充电结束时，系统记录结束时间和最终电表读数，根据实际充电量、分时电价和服务费生成 Bill。总费用计算公式为：

总费用 = 充电费 \+ 服务费

其中充电费按照分时电价分段计算，高峰时段 1\.2 元/度，平峰时段 0\.8 元/度，低谷时段 0\.4 元/度。服务费按充电模式计算，快充 0\.7 元/度，慢充 0\.4 元/度。若充电过程跨越多个时段，系统需要按不同时段对应的实际充电量分别计费。

故障处理模块负责充电桩故障报告、状态变更和重新调度。当充电桩发生故障时，系统将该桩状态改为不可用。如果桩上有正在充电的车辆，系统记录已充电量，将剩余需求重新放入对应队列顶部，并在新的空闲桩出现时优先调度，实现断点续充。管理员确认故障后可以进行维护，修复后解除故障状态并重新开放服务。

后台监控模块面向管理员，提供充电桩状态、车位占用、排队车辆、活跃会话、营收报表、车流分析和平均等待时间等数据。后台监控数据来自 ChargingPile、ChargingSession、Bill、Payment 和 ChargingRequest 等对象的统计结果。

## 五、关键业务流程设计

用户完整充电流程如下：用户登录系统并绑定车辆；到达充电站后选择充电模式和目标电量；系统创建充电请求并加入对应队列；调度模块在有空闲桩时选择车辆并分配充电桩；用户确认后开始充电；系统创建充电会话并实时记录充电状态；充电达到目标、用户手动停止或 BMS 发送充满信号后，系统结束会话；计费模块生成账单；用户查看账单并完成支付。

充电调度流程如下：系统检测到某一充电桩空闲；根据该桩类型选择对应快充或慢充队列；扫描队列中的等待请求；计算每个请求的预计充电时长和预计完成时间；选择预计完成时间最短的请求；更新请求状态为已分配或充电中；更新充电桩状态；通知用户前往对应充电桩。若多个请求预计完成时间相同，则优先选择等待时间较长的请求，避免长期等待。

计费流程如下：开始充电时记录开始时间和初始电表读数；充电过程中可实时估算当前费用；结束充电时记录结束时间和最终电表读数；根据起止时间判断是否跨越高峰、平峰、低谷时段；按不同时间段的电量分别计算电费；根据快充或慢充计算服务费；生成账单明细；用户选择支付方式完成支付；支付成功后更新 Bill 和 Payment 状态。

故障恢复流程如下：系统检测到充电桩故障；将充电桩状态改为故障；若存在正在充电的会话，则停止该会话并记录已充电量；计算剩余充电量；将该请求放回对应队列顶部并设置高优先级；当同类型充电桩空闲时，优先恢复该请求；新桩继续完成剩余充电量；最终正常生成账单。

## 六、接口与对象协作设计

系统主要 Controller 包括 UserController、ChargingController、PileController、SchedulerController、BillController 和 AdminController。

UserController 负责注册、登录、车辆绑定等请求；ChargingController 负责提交充电申请、修改充电目标、取消请求、查询车辆状态；PileController 负责开始充电、结束充电、查看充电桩状态、上电、断电和参数设置；SchedulerController 负责队列状态查询和调度触发；BillController 负责账单查询、详单查询和支付；AdminController 负责后台运营数据查询和故障管理。

业务层主要包括 UserService、ChargingService、SchedulerService、PileService、BillService、PaymentService 和 FaultService。ChargingService 负责请求生命周期管理；SchedulerService 负责队列和调度算法；PileService 负责充电桩状态与硬件适配；BillService 负责费用计算和账单生成；FaultService 负责故障记录、故障确认和故障恢复。

数据访问层主要包括 UserRepository、VehicleRepository、ChargingRequestRepository、QueueRepository、PileRepository、ChargingSessionRepository、BillRepository、PaymentRepository、FaultReportRepository 等。Repository 只负责数据读写，不承担业务规则。

## 七、数据库设计思路

数据库表应围绕业务对象建立。核心表包括 user、vehicle、charging\_request、wait\_queue、queue\_item、charging\_pile、parking\_space、charging\_session、meter\_reading、billing\_tariff、time\_slot\_rate、bill、bill\_detail、payment、fault\_report、maintenance\_record。

其中 user 与 vehicle 是一对多关系；charging\_request 关联 user 和 vehicle；queue\_item 关联 charging\_request；charging\_session 由 charging\_request 转化而来，并关联 charging\_pile；bill 由 charging\_session 生成；payment 与 bill 一对一或一对多关联；fault\_report 关联 charging\_pile；maintenance\_record 关联 fault\_report。

状态字段是系统实现重点。ChargingRequest 可包含 WAITING、ASSIGNED、CHARGING、CANCELLED、COMPLETED、FAILED 等状态；ChargingPile 可包含 IDLE、AVAILABLE、RESERVED、CHARGING、FAULT、MAINTENANCE、OFFLINE 等状态；ChargingSession 可包含 CREATED、CHARGING、PAUSED、RECOVERING、COMPLETED、INTERRUPTED 等状态；Bill 可包含 UNPAID、PAID、OVERDUE、CANCELLED 等状态。

## 八、计费算法设计

计费算法的输入包括开始时间、结束时间、充电总量、充电模式和分时电价规则。由于充电可能跨越多个时段，不能简单用总电量乘以某一个电价，而应按照时间区间进行切分。

算法步骤如下：首先根据开始时间和结束时间切分出若干计费片段；其次根据充电功率或电表读数估算每个片段对应的电量；然后按照该片段所属时段匹配电价；最后累加各片段电费，并计算服务费。

服务费计算公式为：

服务费 = 总充电量 × 模式服务费单价

快充服务费单价为 0\.7 元/度，慢充服务费单价为 0\.4 元/度。最终账单需要保存电费明细、服务费、总费用和支付状态。

## 九、调度算法设计

调度算法以减少车辆完成充电的总时间为目标。系统按充电模式维护快充队列和慢充队列，当某一充电桩空闲时，只扫描同类型队列。

基础算法为最短预计完成时间优先。对于队列中的每个请求，系统根据目标电量、已充电量和充电桩功率计算预计充电时长：

预计充电时长 = 剩余充电量 / 充电桩功率

快充功率为 60kW，慢充功率为 7kW。系统选择预计完成时间最早的请求。如果多个请求相同，则选择等待时间更长的请求，避免饥饿。

为降低实现难度，程序初版可以支持两种策略：FCFS 和 SJF。FCFS 按请求创建时间排序，适合实现简单、结果公平；SJF 按预计充电时长排序，适合提高周转效率。系统可以通过配置项选择当前调度策略。

## 十、异常与边界情况处理

系统需要重点处理以下异常场景。

用户取消请求时，如果请求仍在等待队列中，则直接移除队列项并更新请求状态为 CANCELLED；如果已经开始充电，则不允许取消，只能提前结束充电并结算已产生费用。

用户修改请求时，如果仍在等待队列中，可以修改目标电量并重新计算预计充电时长；如果已经开始充电，只允许提前结束，不允许增加目标电量。

充电桩故障时，如果没有车辆正在充电，则直接将桩状态改为 FAULT；如果存在正在充电会话，则中断会话，记录断点，将剩余需求放入队列顶部并优先恢复。

支付失败时，账单保持 UNPAID 状态，用户可重新支付。若账单逾期未支付，系统可限制用户发起新的充电请求。

管理员修改计费规则时，新规则不应影响已经完成或正在进行的旧会话。建议计费规则设置 effectiveTime，通过生效时间区分新旧规则。

## 十一、程序实现建议

程序实现可以按模块逐步推进，优先完成最小可运行闭环：用户注册与车辆绑定、提交充电请求、进入队列、调度到充电桩、开始充电、结束充电、生成账单、查询账单。这个闭环完成后，再加入故障恢复、后台监控、计费规则管理和支付记录。

推荐优先实现的核心类包括 User、Vehicle、ChargingRequest、ChargingPile、WaitQueue、ChargingSession、Bill、Payment；核心服务包括 ChargingService、SchedulerService、PileService、BillService；核心接口包括 submitChargingRequest、queryQueueState、startCharging、endCharging、generateBill、queryBill。

调度和计费是本系统最能体现设计质量的部分。调度部分应重点保证请求状态、队列状态和充电桩状态同步变化；计费部分应重点保证跨时段计费正确、服务费计算正确、账单明细清晰。

## 十二、总结

本系统的设计核心是用分层架构承载复杂业务，用领域模型表达业务对象，用 Service 层实现调度、计费、故障恢复等核心规则。用户侧流程围绕“提交请求—排队—充电—结算”展开，管理员侧流程围绕“监控—配置—故障处理”展开，系统内部则通过 ChargingRequest、WaitQueue、ChargingPile、ChargingSession 和 Bill 串联起完整业务闭环。

最终程序实现时，应优先保证主流程可运行，再逐步完善调度优化、故障恢复、后台报表和规则配置。只要保证对象职责清晰、状态流转一致、调度算法和计费算法正确，系统即可满足智能充电桩调度计费系统的核心需求。
